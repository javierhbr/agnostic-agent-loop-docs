## Overview

| Level | Scope | Use case | Risk | Phases | Typical timeline |
|---|---|---|---|---|---:|
| **OpenSpec** | single service / team | component changes, features, internal refactors | low | Specify + Deliver only | 1-3 weeks |
| **Platform** | multi-service coordination | contracts, architecture decisions, fan-out | medium | Platform + Assess + Specify + Plan + Deliver | 1-2 weeks |
| **SDD Full** | critical multi-service | payments, auth, PII, regulatory | high/critical | All 5 phases | 3-4 weeks |

**Core boundary rule:** Component repos use OpenSpec ONLY. Platform-side tooling (BMAD, Speckit) stays in the platform repo.

## Level 1: OpenSpec (Component Level)

Use OpenSpec when one team can deliver the change end-to-end without cross-team governance.

### What it gives you

- A lightweight proposal (`proposal.md`) and a task breakdown (`tasks.md`).
- A place to track status, blockages, and contracts (`metadata.yaml`).
- A consistent loop: plan -> implement -> verify -> archive.

### Core commands

```bash
agentic-agent openspec init "Auth Feature" --from requirements.md
agentic-agent openspec list
agentic-agent openspec show <change-id>
agentic-agent openspec status <change-id>
agentic-agent openspec check <change-id>
agentic-agent openspec complete <change-id>
agentic-agent openspec archive <change-id>
```

### Typical directory structure

```text
.agentic/specs/<change-id>/
├── proposal.md
├── tasks.md
└── metadata.yaml
```

### Gate checks (basic)

OpenSpec enforces basic gates:

- Metadata exists (change ID, status, blocked_by)
- No unresolved ADRs blocking the change
- Contracts listed if referencing other services

### Example: Bug fix (low risk)

```bash
cd payments-service/
agentic-agent openspec init "Fix card expiry race condition" \
  --from bug-report.md --type bug
agentic-agent task claim <task-id>
# ... code changes ...
agentic-agent task complete <task-id>
agentic-agent openspec complete card-expiry-fix
agentic-agent openspec archive card-expiry-fix
```

### Use OpenSpec when

- The change is isolated to one service/component.
- No other team must approve contracts or architecture.
- Risk is low (bugs, internal features, refactors).
- Timeline: 1-3 weeks.
- Team-level autonomy (no platform governance needed).

## Level 2: Platform (Architecture / Governance Level)

Use Platform when multiple services are involved or when you need a shared contract baseline before teams implement.

### What it gives you

- An initiative-level view (what ships, when, and who owns what).
- A platform spec (the WHAT), plus fan-out to component teams.
- Gate checks as a governance tool (before fan-out).

### Core commands (illustrative)

```bash
agentic-agent platform init
agentic-agent platform add-feature \
  --name "Subscription Billing" \
  --initiative "ECO-123" \
  --context-pack "cp-v2.1" \
  --fanout "billing-service,payments-service,email-service"
```

### Gate checks (all 5 gates)

Platform specs enforce all 5 SDD gates:

1. **Gate 1 -- Context Completeness:** Every section has Source: line (MCP citations)
2. **Gate 2 -- Domain Validity:** No cross-domain DB access, boundaries respected
3. **Gate 3 -- Integration Safety:** All contracts identified, consumers listed
4. **Gate 4 -- NFR Compliance:** Logging, metrics, tracing, PII, performance specified
5. **Gate 5 -- Ready to Implement:** No blocking ADRs, ACs testable, unambiguous

### Use Platform when

- Multiple services are affected.
- A contract/API decision needs alignment up front.
- Cross-team sequencing and tracking matter.
- Platform team owns governance.
- Medium risk (shared patterns, breaking changes).

## Level 3: SDD Full (Critical Features)

Use SDD Full only when risk demands strict discovery, explicit design, and evidence-based verification.

### What it gives you

- Four role-bounded phases: Analyst -> Architect -> Developer(s) -> Verifier.
- Explicit gates that block progress when quality is insufficient.
- Verification that records observable evidence per acceptance criterion.

### Core commands (illustrative)

```bash
agentic-agent sdd start "Subscription Billing" --risk critical
agentic-agent sdd workflow show <initiative-id>
agentic-agent sdd agents install
agentic-agent sdd gate-check <spec-id>
agentic-agent sdd adr create --title "Blocking decision"
agentic-agent sdd adr resolve <adr-id>
```

### Workflow paths by risk

| Risk | Workflow | Phases | Timeline |
|------|----------|--------|----------|
| Low | Quick | Specify + Deliver | 1-2 days |
| Medium | Standard | Assess (lightweight) + Specify + Plan + Deliver | 3-5 days |
| High/Critical | **Full** | **Platform + Assess + Specify + Plan + Deliver** | **5-10 days** |

### Use SDD Full when

- Risk is high/critical (payments/auth/PII/regulatory).
- Rollback is hard and failure is costly.
- You need an audit trail and explicit sign-off.
- Multiple services affected (5+).
- Blocking decisions needed (ADRs).
- Data consistency is critical.

## Decision Tree

```text
Start: What are you changing?

Single service + low risk?
  -> OpenSpec

Multi-service OR architecture decision?
  Medium risk?
    -> Platform
  High/critical risk?
    -> SDD Full
```

## How the Levels Fit Together

```text
Platform (WHAT + contracts)  ->  fan-out to teams
                                  |
                                  v
OpenSpec per component (proposal + tasks) -> implementation work
                                  |
                                  v
Verifier / evidence (required only when risk warrants it)
```

## Best Practices

### OpenSpec level (Specify + Deliver)

- Write a proposal that your team understands.
- Decompose into 3-10 concrete tasks.
- Use `blocked_by` if waiting on other teams.
- Mark complete only when tested.

### Platform level (Platform + Assess + Specify + Plan + Deliver)

- Write platform spec with clear contracts.
- Use fan-out to distribute to teams.
- Run all 5 gate checks before fan-out.
- Track initiative priority/status.

### SDD Full level (All 5 phases)

- Use for critical features only.
- Get Assess phase buy-in (evidence matters).
- Specify phase must produce `proposal.md` + delta specs.
- Resolve ADRs quickly (they block development).
- Plan phase produces `design.md` + `tasks.md`.
- Deliver phase: Developer runs in parallel, Verifier checks every AC with evidence.

## FAQ

**Q: Can I use OpenSpec and Platform together?**
A: Yes. OpenSpec for component work, Platform for architecture governance. They integrate naturally through spec references.

**Q: Do I have to use all three levels?**
A: No. Use the minimum needed for your risk level: low risk = OpenSpec only; medium risk = Platform only; critical risk = SDD Full (includes Platform + OpenSpec).

**Q: What if I start with OpenSpec and later realize I need Platform/SDD?**
A: You can upgrade anytime. Reference your existing OpenSpec files in Platform specs. Continue to the next phase.

**Q: Can a small team use SDD Full?**
A: Yes. The five phases can be roles for 2-3 people. It takes longer (3-4 weeks) but the evidence trail is valuable for critical features.
