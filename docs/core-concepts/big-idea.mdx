## 00b — Why This Model: What We Took and What We Dropped

<CalloutBox type="amber" label="Selection rule">
  If a methodology component does not change what the agent or engineer writes, it is removed.
</CalloutBox>

This model composes proven ideas from SDD, MCP, ADR practice, Agile/XP, Lean, BMAD, OpenSpec, and SpecKit.
All process lives in Markdown files under version control and review. No process logic lives in extra infrastructure.

| Source | What We Took | What We Dropped |
|---|---|---|
| **SDD** | Spec-before-code, Given/When/Then ACs, embedded exit-gate checklists | Centralized spec registries and snapshot services |
| **MCP** | Platform MCP for shared truth; Component MCP per component for local truth | Router MCP and extra domain/contract MCP split |
| **ADR** | Decisions as first-class artifacts and merge blockers when unresolved | Separate ADR query services |
| **Agile / XP** | Risk-proportional workflows and small handoff artifacts | Mandatory ceremony overhead |
| **Lean** | Eliminate non-output-changing work; logic in files, not servers | Tooling that adds maintenance but no output quality |
| **BMAD** | Four bounded roles, hard-stop handoffs, Architect fan-out for parallel dev | Orchestrator agent and persona naming |
| **OpenSpec** | Mandatory traceability metadata, Spec Graph, source citation, versioned lifecycle | Dedicated spec versioning server |
| **SpecKit** | Templates with embedded gates, Context Pack pinning, typed spec templates via MCP | Separate scaffolding CLI |

## Core Outcome

- Knowledge is governed and versioned.
- Agent behavior is role-bounded and traceable.
- Quality gates are enforced without extra services.
- The system compounds learning after every release through updated specs, ADRs, and patterns.
