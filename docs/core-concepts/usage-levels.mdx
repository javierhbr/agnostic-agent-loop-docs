## Overview

| Level | Scope | Use case | Risk | Typical timeline |
|---|---|---|---|---:|
| **OpenSpec** | single service / team | component changes, features, internal refactors | low | 1-3 weeks |
| **Platform** | multi-service coordination | contracts, architecture decisions, fan-out | medium | 1-2 weeks |
| **SDD Full** | critical multi-service | payments, auth, PII, regulatory | high/critical | 3-4 weeks |

## Level 1: OpenSpec (Component Level)

Use OpenSpec when one team can deliver the change end-to-end without cross-team governance.

### What it gives you

- A lightweight proposal (`proposal.md`) and a task breakdown (`tasks.md`).
- A place to track status, blockages, and contracts (`metadata.yaml`).
- A consistent loop: plan -> implement -> verify -> archive.

### Core commands

```bash
agentic-agent openspec init "Auth Feature" --from requirements.md
agentic-agent openspec list
agentic-agent openspec show <change-id>
agentic-agent openspec status <change-id>
agentic-agent openspec check <change-id>
agentic-agent openspec complete <change-id>
agentic-agent openspec archive <change-id>
```

### Typical directory structure

```text
.agentic/specs/<change-id>/
├── proposal.md
├── tasks.md
└── metadata.yaml
```

### Use OpenSpec when

- The change is isolated to one service/component.
- No other team must approve contracts or architecture.
- Risk is low (bugs, internal features, refactors).

## Level 2: Platform (Architecture / Governance Level)

Use Platform when multiple services are involved or when you need a shared contract baseline before teams implement.

### What it gives you

- An initiative-level view (what ships, when, and who owns what).
- A platform spec (the WHAT), plus fan-out to component teams.
- Gate checks as a governance tool (before fan-out).

### Core commands (illustrative)

```bash
agentic-agent platform init
agentic-agent platform add-feature \
  --name "Subscription Billing" \
  --initiative "ECO-123" \
  --context-pack "cp-v2.1" \
  --fanout "billing-service,payments-service,email-service"
```

### Use Platform when

- Multiple services are affected.
- A contract/API decision needs alignment up front.
- Cross-team sequencing and tracking matter.

## Level 3: SDD Full (Critical Features)

Use SDD Full only when risk demands strict discovery, explicit design, and evidence-based verification.

### What it gives you

- Four role-bounded phases: Analyst -> Architect -> Developer(s) -> Verifier.
- Explicit gates that block progress when quality is insufficient.
- Verification that records observable evidence per acceptance criterion.

### Core commands (illustrative)

```bash
agentic-agent sdd start "Subscription Billing" --risk critical
agentic-agent sdd workflow show <initiative-id>
agentic-agent sdd agents install
agentic-agent sdd gate-check <spec-id>
agentic-agent sdd adr create --title "Blocking decision"
agentic-agent sdd adr resolve <adr-id>
```

### Use SDD Full when

- Risk is high/critical (payments/auth/PII/regulatory).
- Rollback is hard and failure is costly.
- You need an audit trail and explicit sign-off.

## Decision Tree

```text
Start: What are you changing?

Single service + low risk?
  -> OpenSpec

Multi-service OR architecture decision?
  Medium risk?
    -> Platform
  High/critical risk?
    -> SDD Full
```

## How the Levels Fit Together

```text
Platform (WHAT + contracts)  ->  fan-out to teams
                                  |
                                  v
OpenSpec per component (proposal + tasks) -> implementation work
                                  |
                                  v
Verifier / evidence (required only when risk warrants it)
```
