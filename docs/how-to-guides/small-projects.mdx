## Pick a Small-Project Tier

| Tier | Team | Risk | Use | Timeline |
|---|---:|---|---|---:|
| **Tiny** | 1 | low | tasks only | hours to 1 day |
| **Small** | 1-2 | low/medium | OpenSpec + tasks | 1-2 weeks |
| **Medium** | 2-3 | medium | light Platform + OpenSpec | 2-3 weeks |

## Tiny: Bug Fix or Small Refactor (Tasks Only)

### Steps

```bash
agentic-agent task create "Fix login timeout bug"
agentic-agent task claim <task-id>
# ... implement + tests ...
agentic-agent task complete <task-id>
```

### Optional: add a 1-paragraph note (only if you need memory later)

```bash
mkdir -p .agentic/specs/login-timeout-fix
cat > .agentic/specs/login-timeout-fix/README.md << 'EOF'
# Login Timeout Fix

## Problem
Users are logged out after 15 minutes of inactivity, even on idle tabs.

## Solution
Keep refresh token alive with silent renewal on focus.

## Tasks
- [ ] Add focus listener to renew token
- [ ] Test with 30-min idle
- [ ] Update docs
EOF
```

## Small: 1-2 Week Feature (OpenSpec + Tasks)

### Steps

1. Write a short `requirements.md` (problem + acceptance criteria).
2. Initialize OpenSpec.
3. Work task-by-task.
4. Verify acceptance criteria, then complete and archive.

```bash
agentic-agent openspec init "User Dashboard" --from requirements.md --type feature
agentic-agent task claim <task-id>
# ... implement + tests ...
agentic-agent task complete <task-id>
agentic-agent openspec complete <change-id>
agentic-agent openspec archive <change-id>
```

### What to write in `proposal.md`

Keep it short:

```markdown
# User Dashboard

## What
Display user profile, recent orders, preferences in one view.

## Why
Users currently click through 3 pages to see their data.
Goal: reduce time-to-profile-view from 4 clicks to 1.

## Acceptance
- [ ] Dashboard loads in < 500ms
- [ ] Shows profile, orders, preferences
- [ ] Mobile responsive
```

### What to write in `tasks.md`

Decompose into tasks that one developer can complete in 1-3 days:

```markdown
# Tasks

## Backend
1. Create `GET /api/user/dashboard`
   - Acceptance: returns profile + orders + preferences (200)
2. Add caching (5 min TTL)
3. Add integration tests

## Frontend
4. Create Dashboard page
5. Add profile section
6. Add orders section
7. Mobile responsive styles

## QA
8. Test on mobile devices
9. Test with slow network (DevTools throttle)
```

### Minimal gate for small projects (optional)

If you touch external systems (payments/auth/integrations), run a basic check before completion:

```bash
agentic-agent openspec check <change-id>
```

## Medium: 2-3 Services, 2-3 People (Light Platform + OpenSpec)

Use a 1-page platform spec to align on contracts and responsibilities, then let each team implement via OpenSpec.

### Steps

```bash
# platform lead writes the 1-page spec
agentic-agent platform init
agentic-agent platform add-feature \
  --name "Guest Checkout" \
  --initiative "CHECKOUT-2025" \
  --context-pack "cp-v2" \
  --fanout "checkout-service,payments-service"

# each component team implements via OpenSpec
cd checkout-service/
agentic-agent openspec init "Guest checkout flow" --from ../platform-specs/guest-checkout/spec.md
```

## Pattern: The "Light SDD" Workflow

For small teams working on a quick multi-service feature:

```text
┌─────────────────────────────────────┐
│  1. Write 1-page platform spec      │
│     (if multi-service)              │
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│  2. Each team: openspec init        │
│     + simple proposal + tasks       │
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│  3. Implement tasks in parallel     │
│     (no gates, just get it done)    │
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│  4. Quick verification              │
│     "Did we build what was asked?"  │
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│  5. Archive & move to next feature  │
└─────────────────────────────────────┘
```

## Real Example: Small Team Adding Search

**Team:** 2 developers |
**Feature:** Add search to product catalog |
**Risk:** Low (single service) |
**Timeline:** 1 week

### Day 1: Write spec and tasks

```bash
agentic-agent openspec init "Product Search" --from requirements.md
```

### Days 2-6: Build

```bash
# Dev 1 works on backend
agentic-agent task claim backend-elasticsearch
# ... implement ...
agentic-agent task complete backend-elasticsearch

# Dev 2 works on frontend in parallel
agentic-agent task claim frontend-search-box
# ... implement ...
agentic-agent task complete frontend-search-box
```

### Day 7: Finish and deploy

```bash
agentic-agent openspec complete product-search
agentic-agent openspec archive product-search
```

## CLI Shortcuts for Small Projects

### Commands to skip (overkill for small projects)

```bash
agentic-agent specify start ...      # full SDD only
agentic-agent platform init ...      # only if multi-service
agentic-agent specify adr create ... # only if blocking decision
agentic-agent specify gate-check ... # only if high-risk
```

### Commands to use

```bash
agentic-agent openspec init <name>   # Create spec + tasks
agentic-agent task list              # See all tasks
agentic-agent task claim <id>        # Start working
agentic-agent task complete <id>     # Mark done
agentic-agent openspec status <id>   # Check progress
agentic-agent openspec complete <id> # Finish feature
agentic-agent openspec archive <id>  # Clean up
```

## When to Upgrade (Stop Being "Small")

Upgrade to Platform or SDD Full if any of these become true:

- Risk increases: payments, auth, PII, regulatory concerns.
- The task graph grows: more than ~10 tasks or circular dependencies.
- Cross-team ownership: multiple teams own different services.
- Blocking decisions exist: you need ADRs to proceed safely.

Then move to Platform level (skip Assess phase or keep it lightweight) or SDD Full with all 5 phases (if critical risk).

## Small Project Decision Tree

```text
What are you building?

├─ Bug fix or small refactor?
│  └─ Just use: agentic-agent task create + task complete
│     (No spec needed)
│
├─ New feature (1 service, 1-2 weeks)?
│  └─ Use: agentic-agent openspec init
│     └─ Write: 1-paragraph proposal + 5-10 tasks
│     └─ Implement: task by task
│     └─ Complete: openspec complete + archive
│
└─ Multi-service or complex feature?
   └─ Use: Light Platform level
      └─ Write: 1-page platform spec
      └─ Fanout: to component teams
      └─ Each team: openspec init
      └─ Complete: each component + platform priority
```

## FAQ: Small Projects

**Q: Do I really need a spec for a 3-task bug fix?**
A: No. Use `agentic-agent task` only. Specs are for features that take more than 1 week.

**Q: Can I skip tasks.md and just use agentic-agent task?**
A: Yes. Use either `openspec init` (creates proposal + tasks.md) or `agentic-agent task create` directly (skip spec entirely).

**Q: What if I'm a solo developer?**
A: Use `agentic-agent task` to track your work. Skip specs entirely unless you need to remember context later.

**Q: Can I add gates to a small project?**
A: Yes, but it is usually overkill. Just test before marking complete. Gates are for high-risk features.

**Q: What if requirements change mid-project?**
A: Update proposal.md and tasks.md as you go. No approval needed.

**Q: How do I integrate small project specs with larger platform specs?**
A: Reference them: `agentic-agent openspec init "My Feature" --from ../platform-specs/feature-name/spec.md --implements "feature-name"`
