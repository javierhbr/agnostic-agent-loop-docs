## Simple Feature Development

**Scenario:** One developer builds a feature, independent reviewer checks quality, merge when ready.

### Step 1: Create Task and Start Work

```bash
# Developer creates task (or it's already in backlog)
$ agentic-agent task create --title "Add user authentication" \
  --spec-refs auth-feature \
  --scope "internal/auth" "cmd/api" \
  --acceptance "JWT tokens", "Session refresh", "Logout endpoint"

Created task TASK-000123: Add user authentication

# Developer claims the task (auto-creates worktree)
$ agentic-agent task claim TASK-000123

✅ Claimed task TASK-000123: Add user authentication

🔧 Isolated worktree created:
   cd .worktrees/feature/task-000123

📝 When done, run:
   agentic-agent task complete TASK-000123
```

**What happened:**
- Task moved to in-progress
- Worktree created at `.worktrees/feature/task-000123/`
- Dependencies installed (npm/go/python auto-detected)
- Baseline tests ran and passed
- Branch `feature/task-000123` created

### Step 2: Develop in Isolated Worktree

```bash
# Developer enters the worktree
$ cd .worktrees/feature/task-000123

# Develop: build code, write tests
$ cat > internal/auth/jwt.go << 'EOF'
package auth

import (
  "crypto/rsa"
  "time"
)

  privateKey *rsa.PrivateKey
  publicKey  *rsa.PublicKey
}

  // Implementation
}

func (j *JWTManager) VerifyToken(token string) (string, error) {
  // Implementation
}
EOF

# Write tests
$ cat > internal/auth/jwt_test.go << 'EOF'
package auth

import "testing"

func TestCreateToken(t *testing.T) {
  // Test implementation
}

func TestVerifyToken(t *testing.T) {
  // Test implementation
}

func TestExpiredToken(t *testing.T) {
  // Test expired token handling
}
EOF

# Run tests locally in worktree
$ npm test
auth.test.js:
  ✓ CreateToken generates valid token
  ✓ VerifyToken accepts valid token
  ✓ VerifyToken rejects expired token
  ✓ VerifyToken rejects invalid signature

4 passing (120ms)

# Commit changes
$ git add internal/auth/
$ git commit -m "feat: implement JWT authentication

- Create tokens with RS256 signing
- Verify token signature and expiry
- Return user ID from valid tokens
- Tests: 4 passing"

$ git push -u origin feature/task-000123
```

**Worktree benefits:**
- Main branch completely untouched
- Tests run in isolation
- Can commit without affecting other branches
- Full git history on feature branch

### Step 3: Create PR (Auto-Generate from Spec)

```bash
# Developer creates PR from spec + commits
$ agentic-agent pr create --task TASK-000123

Creating PR: feature/task-000123 → main
✅ PR created: https://github.com/myorg/myapp/pull/42

# PR URL printed and saved in task metadata
```

**Generated PR (auto-created):**

```
Title: Add user authentication

Body:
## Summary
JWT-based user authentication with token validation and session management.

## Acceptance Criteria
- [x] JWT tokens signed with RS256
- [x] Session expiry enforced
- [x] Logout endpoint implemented
- [x] Concurrent requests handled

## Commits
- a1b2c3d feat: implement JWT authentication
- e4f5g6h test: add JWT validation tests
- i7j8k9l chore: update dependencies

---
Generated by agentic-agent (task: TASK-000123)
```

### Step 4: Spawn Independent Reviewer

```bash
# Orchestrator creates review task
$ agentic-agent pr review --task TASK-000123 \
  --pr-url https://github.com/myorg/myapp/pull/42

✅ Review task created: REVIEW-42

🔍 Next steps for reviewer:
  1. agentic-agent task claim REVIEW-42
  2. agentic-agent context build --task REVIEW-42
  3. agentic-agent specifyify gate-check auth-feature
  4. agentic-agent validate
  5. Review code, score quality, announce verdict
  6. agentic-agent task complete REVIEW-42

# Reviewer task ID: REVIEW-42
```

### Step 5: Independent Review

```bash
# Reviewer claims the review task (gets own worktree)
$ agentic-agent task claim REVIEW-42

✅ Claimed task REVIEW-42: Review PR #42: Add user authentication

🔧 Isolated worktree created:
   cd .worktrees/feature/task-REVIEW-42

# Reviewer loads context to understand requirements
$ agentic-agent context build --task REVIEW-42 | head -50

# Reviewer runs quality gates
$ agentic-agent specifyify gate-check auth-feature

✅ Gate 2 (Domain): PASS — JWT expiry enforced
✅ Gate 3 (Integration): PASS — no external dependencies
✅ Gate 4 (NFR): PASS — error logging implemented
✅ Gate 5 (Ready): PASS — all 4 AC clear

# Reviewer runs validation rules
$ agentic-agent validate

✅ task_scope: PASS
✅ context_update: PASS
✅ sdd_metadata: PASS
✅ sdd_adr_blocking: PASS
✅ sdd_spec_graph: PASS
... (9/9 PASS)

# Reviewer reads code and spec, checks for risks
# Reviewer scores: 9/10 (excellent, minor doc gap)

$ agentic-agent task complete REVIEW-42 \

📚 Learnings recorded
```

**Reviewer's independent review:**
- Own worktree (separate from builder)
- Can test code independently
- Different perspective catches bugs
- Verdict recorded separately

### Step 6: Developer Completes Task

```bash
# Developer completes task (auto-cleanup)
$ agentic-agent task complete TASK-000123 \
  --learnings "Implemented JWT auth with comprehensive tests. Reviewer approved with score 9/10."

✅ Completed task TASK-000123

✅ Worktree removed: .worktrees/feature/task-000123
```

**Result:**
- Code reviewed independently (9/10)
- All tests passing
- PR ready to merge
- Full commit history preserved
- Worktree cleaned up

## Code Review with Feedback Loop

**Scenario:** First review finds issues, developer fixes them, reviewer re-checks.

### Iteration 1: Initial Review (Score 7/10)

```bash
# Developer: agentic-agent pr create --task TASK-100
# PR #50 created

# Orchestrator: agentic-agent pr review --task TASK-100 --pr-url ...
# REVIEW-50 task created

# Reviewer: agentic-agent task claim REVIEW-50
# Reviewer checks code...

# Reviewer finds issues:
# - HIGH: Missing error handling in database connection
# - MEDIUM: No retry logic for transient failures
# - LOW: Inconsistent variable naming

$ agentic-agent task complete REVIEW-50 \
  --learnings "Code review: 7/10. 1 HIGH risk, 1 MEDIUM risk. Developer must fix before merge.

Findings:
1. HIGH: DB connection error not handled in Connect()
   Location: internal/db/connection.go:45
   Impact: Could crash on connection failure

2. MEDIUM: No retry logic for timeouts
   Location: internal/db/connection.go:60
   Impact: Transient failures will cause immediate failure

3. LOW: Variable naming inconsistent (dbConn vs db_conn)
   Location: internal/db/connection.go throughout
   Impact: Code readability

Action: Developer should fix HIGH + MEDIUM before re-review."

✅ Completed task REVIEW-50
```

### Iteration 2: Developer Fixes Issues

```bash
# Developer sees the feedback and fixes issues
$ agentic-agent task claim TASK-100 --continue

# Developer is back in same worktree
$ cd .worktrees/feature/task-100

# Fix the issues
$ cat > internal/db/connection.go << 'EOF'
package db

import (
  "fmt"
  "time"
)

type ConnectionConfig struct {
  Host     string
  Port     int
  Timeout  time.Duration
  MaxRetry int
}

func (c *ConnectionConfig) Connect() error {
  var lastErr error

  for attempt := 0; attempt < c.MaxRetry; attempt++ {
    err := c.tryConnect()
    if err == nil {
      return nil
    }
    lastErr = err

    // Exponential backoff
    backoff := time.Duration(1<<uint(attempt)) * time.Second
    time.Sleep(backoff)
  }

  return fmt.Errorf("failed to connect after %d attempts: %w",
    c.MaxRetry, lastErr)
}

func (c *ConnectionConfig) tryConnect() error {
  // Implementation with proper error handling
}
EOF

# Run tests to verify fixes
$ npm test
db.test.js:
  ✓ Connect returns error on connection failure
  ✓ Connect retries on transient failure
  ✓ Connect backs off exponentially
  ✓ Connect succeeds on retry

4 passing

# Commit the fix
$ git add internal/db/
$ git commit -m "fix: add error handling and retry logic to database connection

- Handle connection errors gracefully
- Implement exponential backoff for retries
- Add tests for error and retry scenarios
- Consistent variable naming throughout"

$ git push origin feature/task-100
```

### Iteration 2: Re-Review (Score 9/10)

```bash
# Orchestrator creates new review task
$ agentic-agent pr review --task TASK-100 --pr-url ...

# REVIEW-50-2 created (different task ID)

# Reviewer re-checks
$ agentic-agent task claim REVIEW-50-2

# Reviewer verifies fixes:
# - HIGH issue: FIXED ✅
# - MEDIUM issue: FIXED ✅
# - LOW issue: Not critical, can approve anyway

$ agentic-agent task complete REVIEW-50-2 \
  --learnings "Re-review: 9/10. All critical issues fixed. Code is now production-ready. Approve."

✅ Completed task REVIEW-50-2
```

### Developer Completes Task

```bash
$ agentic-agent task complete TASK-100 \
  --learnings "Fixed 1 HIGH + 1 MEDIUM risk from initial review. Re-review passed 9/10."

✅ Completed task TASK-100
```

**Result:**
- Feedback loop successful
- Issues identified and fixed
- Re-review approved
- Code meets quality bar

## Parallel Multi-Agent Development

**Scenario:** Two developers building different features in parallel, both submit PRs same time.

### Time: 10:00 - Both Claim Tasks

```bash
# Developer A
$ agentic-agent task claim TASK-200
✅ Worktree A: .worktrees/feature/task-200/

# Developer B (different shell/machine)
$ agentic-agent task claim TASK-201
✅ Worktree B: .worktrees/feature/task-201/
```

### Time: 10:00-11:00 - Parallel Development

```bash
# Developer A: Building auth module
.worktrees/feature/task-200/
├── internal/auth/
│   ├── jwt.go (in dev)
│   ├── session.go (in dev)
│   └── jwt_test.go (in dev)
└── git branch: feature/task-200

# Developer B: Building metrics module (completely independent)
.worktrees/feature/task-201/
├── internal/metrics/
│   ├── collector.go (in dev)
│   ├── exporter.go (in dev)
│   └── collector_test.go (in dev)
└── git branch: feature/task-201

# Main branch: untouched by either developer
main (clean)
```

**Key advantage:**
- No merge conflicts
- No branch pollution
- Both can commit freely
- Tests run in isolation

### Time: 11:00 - Both Create PRs

```bash
# Developer A
$ agentic-agent pr create --task TASK-200
✅ PR #60 created: feature/task-200 → main

# Developer B (at same time)
$ agentic-agent pr create --task TASK-201
✅ PR #61 created: feature/task-201 → main

# Both PRs submitted simultaneously
# No waiting, no blocking
```

### Time: 11:05 - Both Create Review Tasks

```bash
# Orchestrator
$ agentic-agent pr review --task TASK-200 --pr-url https://.../pull/60
✅ REVIEW-60 created

$ agentic-agent pr review --task TASK-201 --pr-url https://.../pull/61
✅ REVIEW-61 created
```

### Time: 11:10 - Both Reviewers Start

```bash
# Reviewer 1 (reviews auth PR)
$ agentic-agent task claim REVIEW-60
✅ Worktree C: .worktrees/feature/task-REVIEW-60/

# Reviewer 2 (reviews metrics PR, different person/machine)
$ agentic-agent task claim REVIEW-61
✅ Worktree D: .worktrees/feature/task-REVIEW-61/

# Both review in parallel, no conflicts
```

### Time: 11:30 - All Done

```bash
# Reviewer 1
$ agentic-agent task complete REVIEW-60 --learnings "Auth PR: 9/10 APPROVED"

# Reviewer 2
$ agentic-agent task complete REVIEW-61 --learnings "Metrics PR: 8/10 REQUEST_CHANGES (1 MEDIUM risk)"

# Developer A
$ agentic-agent task complete TASK-200

# Developer B (must fix and re-review)
$ agentic-agent task claim TASK-201 --continue
# (fix issue, re-submit)
```

**Result:**
- 2 features developed in parallel
- 0 merge conflicts
- 4 worktrees used, all cleaned up
- 2 PRs reviewed simultaneously
- Full traceability

## Bug Fix with Urgent Review

**Scenario:** Critical bug found in production, fix needs immediate review.

### Step 1: Create Urgent Bug Task

```bash
$ agentic-agent task create --title "CRITICAL: Fix memory leak in cache" \
  --scope "internal/cache" \
  --acceptance "Memory usage constant under load", "No goroutine leak", "All tests pass"

Created task TASK-500: CRITICAL: Fix memory leak in cache
```

### Step 2: Fast Development & Commit

```bash
$ agentic-agent task claim TASK-500
✅ Worktree: .worktrees/feature/task-500/

$ cd .worktrees/feature/task-500

# Quick fix
$ vim internal/cache/cache.go
# Add: defer cache.mu.Unlock()
# Add: cleanup goroutines on Close()

# Run tests
$ npm test
cache.test.js:
  ✓ Memory usage constant (1000 iterations)
  ✓ No goroutine leak
  ✓ Cleanup on shutdown

# Commit
$ git commit -m "fix: prevent memory leak in cache

- Add mutex unlock in all code paths
- Cleanup goroutines on Close()
- Verified: memory constant, 0 goroutine leak"

$ git push -u origin feature/task-500
```

### Step 3: Create PR Immediately

```bash
$ agentic-agent pr create --task TASK-500

✅ PR #999 created: https://github.com/myorg/myapp/pull/999
```

### Step 4: Urgent Review (Express)

```bash
$ agentic-agent pr review --task TASK-500 --pr-url https://.../pull/999
✅ REVIEW-999 created

# High-priority reviewer picks it up
$ agentic-agent task claim REVIEW-999

# Quick review (5 min)
$ agentic-agent specifyify gate-check cache-fix
✅ All gates PASS

$ agentic-agent validate
✅ All validation PASS

# Check: Memory leak fixed?
# YES - defer added, tests verify

$ agentic-agent task complete REVIEW-999 \
  --learnings "URGENT: 10/10 APPROVED. Memory leak fixed. Safe to merge immediately."

✅ Completed REVIEW-999
```

### Step 5: Merge to Production

```bash
# (In actual GitHub, developer merges PR)
$ gh pr merge --squash 999

✅ PR #999 merged to main
```

**Result:**
- Bug identified and fixed in ~15 min
- Reviewed and approved in ~5 min
- Merged to production immediately
- Zero downtime fix

## Summary: Common Patterns

### Pattern 1: Simple Feature (1 Dev + 1 Reviewer)
```
claim → develop → pr create → pr review → complete
Duration: ~2 hours
```

### Pattern 2: With Feedback Loop (1 Dev + 1 Reviewer + 2 iterations)
```
claim → develop → pr create → pr review → fix → pr review → complete
Duration: ~4 hours
```

### Pattern 3: Parallel Features (2+ Devs + 2+ Reviewers)
```
claim₁ → develop₁ ↘
claim₂ → develop₂ ↗ → pr create₁,₂ → review₁,₂ → complete₁,₂
Duration: ~2 hours (same as serial due to parallel work)
```

### Pattern 4: Urgent Bug (1 Dev + 1 Reviewer, express)
```
claim → develop → pr create → express review → complete
Duration: ~20 min
```

## Best Practices

1. **Always claim before starting** — Records timestamp for commit capture
2. **Develop in worktree only** — Main branch stays clean
3. **Commit frequently** — Each commit is captured
4. **Run tests locally** — Catch issues before PR
5. **Create PR after completing dev** — Not mid-development
6. **Wait for independent review** — Don't review your own code
7. **Complete task after review** — Triggers auto-cleanup
8. **Check learnings are recorded** — For project knowledge base
