## The Knowledge System Model

Most engineering orgs treat their codebase as the source of truth. They're wrong. The codebase is the *output* — the product of decisions made upstream in requirements, architecture choices, domain invariants, and integration contracts.

When those upstream inputs are scattered across Slack threads, Jira comments, and engineers' heads, you get:

- AI agents that invent plausible-sounding but incorrect context
- Integration bugs that appear at the worst possible time
- New engineers who take months to become productive
- Architectural drift that compounds until a rewrite is required

**SDD + MCP treats knowledge as a first-class engineering artifact.** Every decision is versioned. Every contract is governed. Every agent has a single, authoritative source to query.

## The Three Laws

<CalloutBox type="amber" label="Law 1 — Spec First">
  **Nothing is implemented without a spec.** Implementation specs live in `docs/specs/` of the component repo. Feature specs live in `openclaw-specs/initiatives/`. The codebase never gets ahead of the spec layer.
</CalloutBox>

<CalloutBox type="teal" label="Law 2 — MCP Always">
  **Agents always call MCP.** Platform MCP serves the constitution, workflow templates, and cross-cutting contracts. Component MCP serves domain invariants and local patterns. Agents never read raw spec files — they always go through the MCP interface.
</CalloutBox>

<CalloutBox type="navy" label="Law 3 — Sequential Gates">
  **Gates are sequential, not optional.** Analyst exits before Architect begins. Architect exits before Developer begins. Developer exits before Verifier begins. Each gate has a checklist that must be green before the next phase starts.
</CalloutBox>

## Why Markdown Agents, Not TypeScript Services

Other methodology frameworks build TypeScript validators, contract registry APIs, and custom tooling. That infrastructure has a maintenance cost — and teams stop using it.

SDD + MCP's overhead is exactly zero additional infrastructure:

| Approach | Overhead | Adoption |
|---|---|---|
| TypeScript gate validators | CI config, npm packages, runtime errors | Brittle — teams route around it |
| External contract registry APIs | Servers to host, APIs to version | Abandoned in 6 months |
| **Markdown agents + MCP** | `.md` files in your repo | Zero — it's just files and prompts |

Gates are checklist items inside Markdown templates. Agents are `.md` files in `.claude/agents/`. MCP servers read the spec directories you already have. **If a methodology component doesn't change what the agent writes, it's eliminated.**
