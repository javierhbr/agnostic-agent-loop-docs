# Specify. *Validate.* Execute.

A practical guide to **Spec-Driven Development** with two MCP servers, four Markdown agents, and zero overhead services — the enterprise model for teams that need alignment, traceability, and zero tribal knowledge.

<CalloutBox type="amber" mono>
  "Nothing is implemented without a validated spec backed by governed context." — Core Principle, v3.0
</CalloutBox>

<h4>00 — THE PROBLEM</h4>

## Why SDD + MCP?

Every team hits the same wall. Requirements live in Slack, decisions get lost in PR comments, AI agents invent context they don't have, and contract registries (as servers) add infrastructure that nobody maintains. Teams skip the process or drown in it.

<CalloutBox type="navy" label="The answer:">
  Two MCP servers that serve governed knowledge. Four Markdown agents that carry the process. Zero overhead services — gates live as checklists inside templates, not in TypeScript validators. If a methodology component doesn't change what the agent writes, it's eliminated.
</CalloutBox>

## Core Principle

> Your organisation is a **knowledge system**, not just a code system. Code is the output — specs, contracts, and decisions are the inputs that determine whether the code is right.

The model has three non-negotiable rules:

1. **Nothing is implemented without a spec.** Every change starts in `openclaw-specs/` or the component's own `docs/specs/` directory.
2. **Agents always call MCP.** Platform MCP for constitution and contracts. Component MCP for domain invariants. Never read raw files directly.
3. **Gates are sequential.** No agent moves to the next phase until its exit gate checklist is complete.
